STHLM Triathlon — Implementationsplan

Scrum · Walking Skeleton · 8 sprintar · ~16 veckor

🦴 Walking Skeleton-principen

Vi börjar med ett "walking skeleton" — ett minimalt men fullt fungerande system som kopplar ihop alla arkitekturella lager: frontend → backend → databas → hosting. Det är inte komplett, men det lever, kan driftsättas och tar ett steg från ände till ände. Allt därefter bygger ovanpå det skelettet, sprint för sprint.

📐 Scrum-format

Varje sprint är 2 veckor. Varje sprint levererar fungerande, testbar mjukvara. Vi följer prioriteringsordningen: kärna först (fungerande data och flöden), sedan schema och konfliktlogik (det unika värdet), sist import, mobilpolish och lansering. Säkerhetsgranskning sker löpande och summeras i sista sprinten.

🛠 Teknikstack

Next.js 14 TypeScript Tailwind CSS Supabase PostgreSQL Vercel

One codebase, one language, managed infrastructure. GitHub → Vercel CD pipeline från dag ett.

📅 Tidslinje

8 sprintar × 2 veckor = 16 veckor (ca 4 månader). Tävlingen är 9 augusti 2026. Mjukvaran bör vara produktionsklar senast slutet av juni 2026 för att ge tid till testning och onboarding av funktionärer i juli.

Sprint 0 — Walking Skeleton
🦴 Walking Skeleton — Det första skelettet som lever
Sprint 0 · Vecka 1–2
Infrastruktur, Auth & Roller
Målet är inte funktioner — det är ett fungerande, driftsatt system som bevisar att alla lager fungerar tillsammans. En tävlingsledare kan logga in och se en tom dashboard. En funktionär kan logga in och se "Välkommen". Det är tillräckligt för att bygga allt annat på.
2 veckor 🌐 Live på Vercel
🏗 Setup & Infrastruktur
  • Skapa Next.js-projekt med TypeScript + Tailwind
  • Koppla till GitHub-repot (jonterylle/NewInventions)
  • Sätta upp Vercel + automatisk deploy vid push
  • Skapa Supabase-projekt (databas + auth)
  • Konfigurera miljövariabler (.env) säkert
  • Brand kit i Tailwind-konfiguration (färger, font)
🔐 Auth & Databas
  • Magic link-inloggning via Supabase Auth (e-post)
  • Databasschema: users, roles, volunteers (tomma tabeller)
  • Row-Level Security (RLS) i Supabase — grundregler per roll
  • Middleware: skydda rutter baserat på inloggning och roll
  • Skapa testanvändare: tävlingsledare, sektionsledare, funktionär
📄 Första sidor (skelett)
  • Inloggningssida med STHLM Triathlon-design
  • Tom dashboard — tävlingsledare ser "0 funktionärer"
  • Tom startsida — funktionär ser "Välkommen, [namn]"
  • Navbar + sidopanel-layout (återanvänds alla admin-sidor)
  • 404-sida och felsidor
  • Mobil-navigation (hamburgermeny)
Definition of Done: App lever på Vercel-URL Push till GitHub → auto-deploy fungerar Magic link-login fungerar Tre roller loggar in och ser rätt vy DB-anslutning verifierad

Sprint 2 — Sektioner & Uppdrag
Sprint 2 · Vecka 5–6
Dynamiska sektioner & Uppdrag
Tävlingsledaren kan skapa och konfigurera sektioner (inklusive tvärsektionella). Funktionärer kan tilldelas uppdrag med starttid, sluttid och plats — en funktionär kan ha flera uppdrag.
2 veckor 🗂 Sektioner & pass
🗄 Databas

  • Tabell: section_spans (sektion ↔ annan sektion)
  • Tabell: assignments (volunteer_id, section_id, role, location, start_time, end_time)
  • FK-relationer och index för tidsfrågor
🖥 Sektionshantering

  • Skapa / redigera sektion (namn, ikon, färg, ledare)
  • Toggle: tvärsektionell + välj vilka sektioner den spänner över
  • Ta bort sektion (om inga aktiva funktionärer)
  • Ordna om sektioner (drag eller piltangenter)
📋 Uppdrag per funktionär
  • Lägg till uppdrag på en funktionär (sektion, plats, tider)
  • En funktionär kan ha flera uppdrag
  • Uppdrag visas som lista på funktionärens profilsida
  • Redigera och ta bort enskilt uppdrag
  • 🔐 Säkerhetsgranskning av Sprint 2-kod
Definition of Done: Sektioner kan skapas och redigeras dynamiskt Tvärsektionell-flagga fungerar Funktionär kan ha flera uppdrag med tider Sektionsledare kopplad till sektion
Sprint 3 — Tidslinje & Konfliktdetektering
Sprint 3 · Vecka 7–8
Gantt-tidslinje & Automatisk konfliktdetektering
Sektionsledaren kan nu se sitt sektionsschema som ett Gantt-diagram. Systemet detekterar automatiskt överlappande uppdrag och markerar dem tydligt. Det här är appens kärna — det unika värdet.
2 veckor 📅 Tidslinje live
⚙️ Konfliktlogik (backend)
  • Algoritm: detektera överlappande uppdrag per funktionär
  • Körs automatiskt vid varje sparing av uppdrag
  • Tabell: conflicts (assignment_1, assignment_2, overlap_start, overlap_end, status, approved_by)
  • API-endpoint: hämta konflikter per sektion
📅 Gantt-tidslinje
  • Tidslinje-komponent: en rad per funktionär, staplar per uppdrag
  • Konflikter markeras med rött + röd overlay-zon
  • Godkända konflikter visas med grön streckad kant
  • Klick på stapel öppnar detalj-popup
  • Filter: visa alla / visa bara konflikter
👔 Sektionsledarvy
  • Sektionsledare ser sin sektion i tidslinjen
  • Konflikt-kort ovanför tidslinjen med åtgärdsknappar
  • Kan godkänna konflikter i sin sektion
  • Kan ej se eller redigera andra sektioner
  • 🔐 Säkerhetsgranskning Sprint 3
Definition of Done: Gantt-tidslinje renderas korrekt Konflikter detekteras automatiskt Konflikter visas visuellt Sektionsledare ser bara sin sektion
